# 生成后验证

生成测试代码后，必须验证编译和执行两项内容。

## 编译验证

1. 根据检测到的构建工具运行编译命令：
   - **Maven**：`mvn test-compile -q`
   - **Gradle**：`gradle testClasses -q`
   - **npm/yarn**：`npx tsc --noEmit`（TypeScript）
   - **pytest**：`python -m py_compile <测试文件>`（Python）

2. 如果编译失败：
   - 仔细阅读错误信息
   - 修复测试代码（**不得修改生产代码**）
   - 重新运行编译
   - **最多 5 次尝试**——如果仍然失败，停止并告知用户

3. 常见编译问题：
   - 构造器参数错误（重新读取依赖类）
   - 缺少 import
   - 类型不匹配
   - 缺少断言的静态 import
   - Mockito/AssertJ 版本不兼容

## 测试执行验证

1. 运行生成的测试类：
   - **Maven**：`mvn test -Dtest={测试类名} -q`
   - **Gradle**：`gradle test --tests {测试类名} -q`
   - **npm/yarn**：`npx jest {测试类名} --silent`
   - **pytest**：`python -m pytest {测试文件} -q`

2. 如果测试失败：
   - 仔细阅读失败输出
   - 修复测试以匹配生产代码的实际行为
   - **不得修改生产代码**
   - 重新运行测试
   - **每个测试最多 3 次尝试**——如果仍然失败，移除该测试并告知用户

3. 如果测试失败揭示了生产代码中的潜在 Bug：
   - 修复测试以匹配当前行为
   - 添加注释：`// 注意：当前行为可能存在 Bug — {描述}`
   - 告知用户该潜在问题

## 最终报告

验证完成后，向用户报告：
- 测试总数
- 通过/失败状态
- 被移除的测试及原因
- 发现的潜在 Bug
- 后续建议（如有）

## 模块模式：批量验证

当为模块生成多个测试类时，采用批量验证策略：

### 批量编译

1. 生成所有测试文件后，运行一次全量编译
2. 如果编译失败：
   - 分析错误，按类隔离编译问题
   - 逐个修复，每次修复后重新全量编译
   - **全量编译最多 5 次尝试**
3. 编译策略优势：一次编译可发现所有类的编译问题，避免逐类编译的开销

### 逐类执行

1. 编译通过后，按拓扑顺序逐个运行测试类
2. 每个类独立执行和验证，单个类的失败不影响其他类
3. 每个类的执行结果立即记录
4. 执行进度：`[{当前序号}/{总数}] {ClassName}Test — {通过数}/{总数} 通过`

### 失败隔离策略

- 单个测试类的失败不会阻塞其他类的执行
- 单个测试方法的失败（3 次修复后仍失败）会被移除，不影响同类的其他方法
- 每个类有独立的修复尝试计数
