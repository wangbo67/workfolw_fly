# 已有测试感知

在生成测试之前，检查已有内容。匹配项目的测试约定，避免重复覆盖。

## 生成前：检查已有测试

1. **查找目标的已有测试类：**
   - 在测试目录中搜索 `{ClassName}Test` 或 `{ClassName}Tests`
   - 如果找到了，在生成任何内容之前完整阅读它

2. **如果发现了已有测试类：**
   - **不要**创建新的测试类——在已有类中补充缺失的测试方法
   - 保留已有的测试结构、import 和辅助方法
   - 遵循已有模式（命名、断言风格、初始化方式）
   - 只添加尚未覆盖的行为测试

3. **如果没有发现已有测试类：**
   - 扫描同包下 2-3 个相邻测试类，学习项目约定
   - 匹配风格：import 顺序、断言库、命名模式、注释风格

## 需要从已有测试中匹配的内容

- **断言库**：如果项目使用 Hamcrest，不要切换到 AssertJ（反之亦然）
- **测试数据模式**：如果项目有 `TestDataFactory` 或 Builder，使用它们
- **基础测试类**：如果测试继承了 `BaseTest` 或 `AbstractIntegrationTest`，遵循该模式
- **静态 import 风格**：匹配项目导入断言方法的方式
- **注释风格**：`// given / when / then` 还是 `// arrange / act / assert`

## 错误示例

```java
// 错误：已有测试类存在时又创建了新的
// 文件：UserServiceTest.java（新建 — 重复！）
@ExtendWith(MockitoExtension.class)
class UserServiceTest {
    // 10 个测试方法，其中 5 个与已有文件重复
}
```

## 正确示例

```java
// 正确：只在已有文件中补充缺失的测试
// 文件：UserServiceTest.java（已有 — 追加）
@ExtendWith(MockitoExtension.class)
class UserServiceTest {
    // ... 已有测试原样保留 ...

    @Test
    void updateUser_invalidEmail_throwsValidationException() {
        // ...
    }
}
```

## 检查清单

在编写任何测试代码之前，确认：
- [ ] 已搜索目标的已有测试类
- [ ] 已阅读已有测试，了解已覆盖的内容
- [ ] 已从相邻测试文件中识别项目测试约定
- [ ] 已确认哪些行为仍需要测试覆盖

## 模块模式：批量已有测试检查

当目标为模块/包时，需要批量检查所有目标类的已有测试状态：

### 扫描策略

1. 根据目标类列表，在测试目录中批量搜索对应的 `{ClassName}Test` 或 `{ClassName}Tests`
2. 对搜索结果分类：
   - **已有测试**：标记为 "✅ 需补充"，阅读已有测试内容
   - **无测试**：标记为 "❌ 需新建"

### 覆盖率差距分析

对于已有测试的类，分析缺失的测试覆盖：

1. 提取已有测试方法名，识别已覆盖的方法和场景
2. 与第 2 步推导的用例对比，识别缺失的用例
3. 只补充缺失的用例，避免重复

### 项目风格学习

- 如果模块内大部分类无已有测试，选择同项目内任意 2-3 个相邻测试类学习风格
- 如果部分类已有测试，优先从这些已有测试中学习风格
- 确保同一模块内所有生成的测试保持一致的风格

### 模块模式额外检查

- [ ] 已批量扫描所有目标类的已有测试
- [ ] 已对每个类标记测试状态（已有/新建）
- [ ] 已分析已有测试的覆盖差距
- [ ] 已确定统一的项目测试风格
